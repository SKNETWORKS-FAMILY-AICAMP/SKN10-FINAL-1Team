# 📡 가상 통신사 AI 시스템 도입 타당성 보고서 (2025)

## 1. 🔐 외부 LLM 사용 제한
- 통신사는 개인정보보호법, 전기통신사업법 등으로 인해 외부 LLM(API 기반) 사용이 금지되는 경우 다수
- 민감한 고객정보, 망 제어 데이터 등 외부 API 전송 금지
- OpenAI, Google, Anthropic API 사용 불가 → 내부 구축형 LLM/RAG 필수

### ✅ 해결책:
- 자체 GPU 서버에 LLM + RAG 시스템 구축
- LangChain, LlamaIndex 등으로 사내 PDF 문서 검색/요약
- 완전한 사내망 격리 환경에서 운영

---

## 2. 📚 사내 문서 검색 자동화의 필요성
- 수만 개에 이르는 기술문서, 요금제, 절차문서, 정책 문서가 사내에 존재
- 검색은 키워드 기반, 문서 분산 → 정보 단절 심각
- “망 개통 절차”, “O-RAN 설정 위치” 등 반복 질문 발생

### ✅ 해결책:
- 벡터 DB 기반 PDF RAG 시스템 구축
- Notion AI Q&A처럼 "회사 전체의 기억"을 질문 기반으로 즉시 제공

---

## 3. 🧠 Git 코드 기반 설명 자동화
### ✅ 이유 1: 레거시 코드 이해의 어려움
- 네트워크 제어 시스템, CRM 등 코드량 방대
- 주석 부족, 함수 명 혼란, 문서 부재

### ✅ 이유 2: 인수인계 및 유지보수
- 이직/이동 시 코드 맥락 상실
- 커밋 내역, 함수 의도, API 사용 예시 등에 대한 설명 필요

### ✅ 해결책:
- CodeT5+, StarCoder, Code Llama 등 오픈소스 모델 활용
- GitHub 연동 후 자동 요약 / 함수 설명 / PR 요약 생성 가능
- Copilot X, Claude 등의 기능을 사내망용으로 내재화

---

## 4. 👥 접근제어 필요성 (RBAC/ABAC)
- 모든 문서/코드를 누구나 검색 가능하면 보안사고 위험
- 부서, 등급, 인가 수준에 따라 접근 제한 필요

### ✅ 해결책:
- RBAC: 역할 기반 필터링 (ex. "재무팀만 재무문서 열람")
- ABAC: 속성 기반 제어 (ex. "문서 등급 3 이상은 등급 3 이상만 열람 가능")
- LlamaIndex, Pinecone, Haystack 등에서 필터 쿼리 가능

---

## ✅ 도입 효과 요약

| 항목             | 도입 전                              | 도입 후                              |
|------------------|--------------------------------------|---------------------------------------|
| 문서 검색        | 키워드 검색, 수동 탐색                | 자연어 질문 + 인용 + 요약 자동 제공  |
| 코드 이해        | 직접 리딩, 비효율적                  | 함수 설명 자동 요약, 커밋 문서 생성  |
| 접근 보안        | 문서 공유 어려움 or 과도한 차단     | 세분화된 접근 제어 + 유연한 검색 가능|
| 지식 온보딩      | 문서 구조 파악 소요                   | "망 개통 절차 알려줘" 식의 질문 가능 |

## 🧠 결론
> 가상의 통신사처럼 보안이 중요한 기업은 외부 LLM 사용이 불가하므로,
> RAG 기반 사내 GPT + Git 코드 설명기를 도입해야 한다.
>
> 이는 조직 내 지식 검색, 코드 유지보수, 보안 제어의 핵심 인프라가 될 것이다.
